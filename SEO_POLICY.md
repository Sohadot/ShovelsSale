# SEO Policy — ShovelsSale.com

## Purpose

This document defines the SEO policy for ShovelsSale.com.

ShovelsSale.com does not pursue low-quality traffic, keyword stuffing, mass content inflation, or generic blog growth. SEO exists to strengthen the asset’s authority as a reference system around the Shovel Economy.

SEO must serve the doctrine.

The goal is not only discovery. The goal is reference authority.

---

## 1. Strategic SEO Positioning

ShovelsSale.com should become associated with the following semantic territory:

- Shovel Economy
- Picks-and-Shovels Economy
- Infrastructure Assets
- Market Infrastructure
- Gold-Rush Infrastructure
- Enabler Economics
- Infrastructure Capture
- Strategic Digital Assets
- Tooling Layer
- Miner / Shovel / Gatekeeper classification
- Sovereign digital asset stewardship

Every new SEO decision must strengthen this semantic territory.

---

## 2. Core Site Architecture

The core architecture must remain coherent:

- `/manifesto/` — doctrine and philosophical foundation
- `/framework/` — conceptual system and classification logic
- `/scanner/` — interactive analysis and applied classification
- `/dispatch/` — analytical archive and observed cases
- `/blog/` — long-form reference content
- `/guide/` — practical orientation and user navigation

These sections should not compete with each other. They should reinforce each other.

The preferred authority flow is:

Manifesto → Framework → Scanner → Dispatch → Blog → Guide

From a user journey perspective:

Why this matters → How the system works → How to classify → What cases prove it → How to apply it

---

## 3. URL Standards

All URLs must be:

- lowercase
- hyphenated
- readable
- non-abbreviated unless the abbreviation is industry-standard
- stable over time
- free from unnecessary extensions where possible

Approved examples:

- `/framework/`
- `/scanner/`
- `/dispatch/`
- `/blog/domain-names-as-digital-shovels/`
- `/blog/ai-infrastructure-shovels/`

Rejected examples:

- `/Framework/`
- `/aiTools2026/`
- `/post1.html`
- `/new-page-final/`
- `/seo-keyword-best-tools-money/`

A URL is a governance artifact. Changing a URL changes the structure of the asset and must be treated carefully.

---

## 4. Metadata Standards

Every indexable HTML page must include:

- `<title>`
- `<meta name="description" content="..." />`
- `<link rel="canonical" href="https://shovelssale.com/..." />`
- `<meta name="robots" content="index, follow" />`

Recommended Open Graph metadata:

- `og:title`
- `og:description`
- `og:type`
- `og:url`
- `og:image`

Recommended Twitter/X metadata:

- `twitter:card`
- `twitter:title`
- `twitter:description`
- `twitter:image`

Metadata must be written for accuracy, trust, and authority. It must not be clickbait.

---

## 5. Title Standards

Page titles should be specific, durable, and aligned with the asset’s reference role.

Preferred patterns:

- `The Shovel Economy Framework — ShovelsSale.com`
- `Shovel Economy Scanner — Classify Infrastructure Value`
- `Dispatch 001 — [Topic] | ShovelsSale.com`
- `[Article Title] — ShovelsSale.com`

Title rules:

- Avoid exaggerated claims.
- Avoid generic marketing slogans.
- Avoid keyword stuffing.
- Include ShovelsSale.com where appropriate.
- Preserve institutional tone.

---

## 6. Meta Description Standards

Meta descriptions should explain the page’s purpose clearly.

A good meta description should:

- describe the page accurately
- include relevant conceptual language
- avoid hype
- avoid vague promises
- support trust and click clarity

Rejected patterns:

- `Best tools to make money fast.`
- `Amazing secrets nobody tells you.`
- `Click here to discover the ultimate trick.`

Approved pattern:

- `A reference framework for classifying whether a company, product, domain, or market position belongs to the extraction layer, infrastructure layer, or control layer of the Shovel Economy.`

---

## 7. Sitemap Policy

`sitemap.xml` must include all public indexable pages.

Sitemap rules:

- Include canonical public URLs only.
- Exclude temporary, private, test, draft, report, and local files.
- Exclude non-indexable utility artifacts.
- Keep URL structure clean and stable.
- Use `<lastmod>` only when there is a meaningful update.
- Avoid artificial daily churn in `<lastmod>`.

Preferred future behavior:

- `lastmod` should derive from the file’s actual meaningful update time or explicit metadata.
- Automated sitemap generation should fail loudly if a page cannot be interpreted correctly.
- Sitemap changes should be reviewed when large URL changes occur.

---

## 8. Robots Policy

`robots.txt` should:

- allow crawling of public indexable content
- reference the sitemap
- avoid blocking important public sections
- protect private or non-public paths where applicable

Required sitemap reference:

`Sitemap: https://shovelssale.com/sitemap.xml`

---

## 9. Internal Linking Policy

Internal links must reflect conceptual architecture.

Required linking behavior:

- Manifesto should link to Framework.
- Framework should link to Scanner.
- Scanner should link back to Framework and relevant explanatory content.
- Dispatch entries should link to Framework and Scanner where relevant.
- Blog posts should link to relevant doctrine, framework, scanner, or dispatch pages.
- No important page should be orphaned.

Internal linking should create a reference graph, not random navigation.

---

## 10. Content Expansion Policy

New content may be added only when it strengthens the asset.

Approved content types:

- Doctrine pages
- Framework pages
- Classification guides
- Dispatch entries
- Case studies
- Market maps
- Tool explanations
- Reference articles
- High-quality glossary nodes, if they are integrated

Rejected content types:

- Thin SEO pages
- Generic listicles
- Unverified trend summaries
- Keyword-only articles
- AI filler articles
- Pages created only to increase URL count
- Affiliate-first articles with weak reference value

Every new content page must answer:

1. What authority does this page add?
2. Which existing concept does it strengthen?
3. Which page links to it?
4. Which page should it link back to?
5. Why should this page exist on ShovelsSale.com specifically?

---

## 11. Dispatch SEO Policy

Dispatch entries should be treated as analytical records, not news spam.

Each dispatch should include:

- clear topic
- date or sequence number
- strategic classification
- connection to the Shovel Economy framework
- internal links to relevant framework or scanner pages
- durable analysis rather than temporary commentary

Dispatch content should age as archive value.

---

## 12. Blog SEO Policy

Blog posts should be long-term reference content.

A blog post should not be published unless it has:

- clear thesis
- original framing
- strong relation to Shovel Economy
- internal links
- metadata
- canonical URL
- durable value

Blog posts should avoid generic trend content unless the trend is analyzed through the Shovel Economy framework.

---

## 13. Structured Data Policy

Structured data should be added carefully and only when accurate.

Potential schema types:

- `WebSite`
- `Organization`
- `Article`
- `BlogPosting`
- `BreadcrumbList`
- `FAQPage`
- `SoftwareApplication`
- `CreativeWork`

Schema must not claim features, ownership, reviews, ratings, or commercial status that are not real.

Structured data should support discoverability without exaggeration.

---

## 14. Image SEO Policy

Images should support trust and clarity.

Image rules:

- Use descriptive file names.
- Avoid generic placeholder names.
- Avoid broken image references.
- Provide meaningful alt text where images convey content.
- Optimize file size.
- Keep Open Graph images stable for important pages.

Approved examples:

- `shovel-economy-framework-map.jpg`
- `infrastructure-layer-diagram.png`

Rejected examples:

- `image1.png`
- `final-final-banner.jpg`
- `placeholder.jpg`

---

## 15. SEO Integrity Failures

The following are considered SEO policy violations:

- Missing title
- Missing meta description
- Missing canonical
- Broken internal links
- Missing referenced assets
- Orphaned public pages
- Sitemap excluding important public pages
- Sitemap including non-public or test files
- Artificial lastmod churn
- Thin content
- Keyword stuffing
- Affiliate-first content without authority value
- Generic AI-generated filler
- Conceptual downgrade
