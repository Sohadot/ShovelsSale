## 1. Purpose of Dispatch Intelligence

Dispatch pages on ShovelsSale.com are structural intelligence dossiers.

Their purpose is to document, with source-backed evidence and explicit confidence labeling, the structural market-layer position of companies, platforms, products, infrastructure layers, and digital assets — as understood through the Shovel Economy Framework.

A Dispatch dossier explains the market power that exists **beneath visible market waves**: the infrastructure capture, control-layer authority, switching-cost compounding, and dependency architecture that determine durable structural value — regardless of who wins the visible competition above.

Dispatch pages are not:
- News articles or market commentary
- Generic company profiles
- Investment notes, price targets, or financial recommendations
- Hype pieces or narrative-driven summaries
- AI-generated filler dressed as analysis
- Short summaries of publicly available information

Dispatch pages are:
- Structured intelligence records built from source-governed facts
- Applied structural analysis using the Shovel Economy Framework
- Governed documents with explicit uncertainty and confidence labeling
- Reference-grade assets intended to remain accurate and useful over a multi-year horizon
- Part of the durable intellectual infrastructure of ShovelsSale.com

Every Dispatch dossier strengthens the asset's position as a sovereign-grade, trusted, reference-grade intelligence resource.

## 2. Required Dispatch Structure

Every new Dispatch page must include the following sections, in order. Sections may be expanded but not omitted. Where evidence is insufficient, the section must state this explicitly rather than be removed.

### 2.1 Executive Intelligence Brief

A 3–5 sentence structural summary for analysts and operators who need the core classification and key structural argument in under 60 seconds.

Must state:
- The actor's name and market context
- The primary structural classification (Miner, Shovel, Gatekeeper, Hybrid, Unclear)
- The dominant structural reason for that classification
- The primary confidence level (High, Moderate, Watch Signal)
- Any critical uncertainty the analyst must know before using this classification

Must not contain investment language, price references, or hype framing.

### 2.2 Classification Summary

A structured record of the scanner-derived classification, including:

| Field | Value |
|---|---|
| Actor | |
| Actor Type | Company / Platform / Product / Domain / Infrastructure |
| Market Wave | |
| Primary Classification | Miner / Shovel / Gatekeeper / Hybrid / Unclear |
| Secondary Classification | (if Hybrid) |
| Confidence Level | High / Moderate / Watch Signal / Low |
| Scanner Scores | Miner: X/100 · Shovel: X/100 · Gatekeeper: X/100 |
| Classification Date | |
| Classification Version | |
| Signal Source | Preset / Manual analysis |

### 2.3 Historical Position

A source-backed timeline of how the actor's structural position has evolved.

Requirements:
- Minimum 3 historical reference points
- Each claim must be source-attributed (see Source Governance)
- Structural transitions must be clearly explained, not just listed as dates
- Distinguish between confirmed history and analyst interpretation of that history

### 2.4 Current Structural Position

A detailed analysis of the actor's structural position at the time of writing.

Must address:
- Infrastructure capture: what layer does this actor operate in?
- Control mechanisms: what access, standards, or routing does this actor control?
- Switching cost architecture: how embedded is this actor in dependent workflows?
- Dependency compounding: does dependency grow, shrink, or remain stable over time?
- Replacement difficulty: how many credible alternatives exist at what cost?

Each substantive claim in this section must be source-attributable or explicitly flagged as analyst interpretation.

### 2.5 Future Projects and Forward Watch Layer

A source-governed forward intelligence section documenting publicly announced developments.

**Gate:** No future project may be included unless it meets all three requirements:
1. Publicly announced by the actor or a regulatory body
2. Source-backed with a cited reference (date-checked)
3. Clearly distinguished from speculation or analyst inference

When no qualifying future projects exist, this section must state: "No qualifying publicly-announced future projects at time of writing. See Source Governance rules."

Do not invent forward signals. Watch signals must be clearly labeled as signals, not conclusions.

### 2.6 Dependency Map

A structured description of what depends on this actor and what this actor depends on.

Two-part structure:

**Upstream dependencies** (what this actor relies on):  
List known dependencies this actor has on other infrastructure layers, suppliers, standards bodies, or platforms. Source each where available.

**Downstream dependencies** (what relies on this actor):  
List known market participants, sectors, or systems that depend on this actor. Include operational dependency types (data, access, compute, payment, identity, distribution, etc.).

Use this section to establish whether the actor is a node in critical infrastructure or an isolated participant.

### 2.7 Replacement Difficulty

A structured assessment of how difficult it would be to replace this actor.

Assess:
- Number of credible alternatives (0 = monopoly infrastructure, 1–2 = oligopoly, 3+ = contested)
- Migration cost type: technical, regulatory, contractual, data-portability, workflow
- Migration timeline: weeks, months, years, decade-scale
- Evidence of attempted replacement (historically) and outcomes

Label the overall replacement difficulty as: **Very High / High / Moderate / Low / Unknown**.

### 2.8 Control-Layer and Shovel-Layer Analysis

A focused analytical section on the actor's structural authority.

Address separately:
- **Shovel-layer signals**: does this actor enable many participants regardless of outcome?
- **Control-layer signals**: does this actor define access, standards, or routing that others must pass through?
- **Compounding dynamics**: does the actor's structural position strengthen as the market matures?
- **Fragility factors**: what could erode this structural position?

This section should connect directly to the scanner signal scores from Section 2.2.

### 2.9 Scanner Interpretation

An explanation of how the scanner signals translate into the classification for this specific actor.

Must address:
- Which signals are driving the primary classification
- Which signals are contested or uncertain
- How a change in key signals would affect the classification
- Whether the Shovel Scanner preset is available for this actor and what values it uses

This section makes the classification legible, auditable, and improvable.

### 2.10 Source-Governed Fact Table

A structured table of key factual claims used in this dossier.

Each row:

| Claim | Source | Source Tier | Date Checked | Confidence |
|---|---|---|---|---|
| | | T1/T2/T3/T4 | | |

Tier definitions: see Section 3 (Source Governance).

Minimum 5 fact table entries for any published Dispatch. Claims used without a source entry must be marked as analyst interpretation, not fact.

### 2.11 Analyst Use Cases

3–5 specific use cases describing who would use this Dispatch and for what operational purpose.

Examples of valid use cases:
- A strategist evaluating infrastructure exposure in a sector
- A builder assessing lock-in risk before adopting a platform
- An operator building a dependency map for business continuity planning
- An analyst calibrating sector-level structural classification

Use cases must be practical, not promotional. They must not imply investment activity.

### 2.12 Limits and Uncertainty

An explicit statement of what this dossier does not cover and where uncertainty remains.

Must address:
- Data limitations (what the analyst could not source)
- Classification limits (what structural signals are missing or contested)
- Temporal limits (when this analysis may become stale)
- Scope limits (what adjacent topics were intentionally excluded)
- The standard not-investment-advice declaration:

> This Dispatch dossier is a structural classification record. It is not investment advice, a financial recommendation, or a valuation. No claim in this document should be used as the sole basis for any financial decision.

### 2.13 Related Dispatch Cases

Links to 2–5 related Dispatch cases with a one-sentence structural connection.

Format:
- [Dispatch NNN — Actor Name](/dispatch/NNN.html): One sentence describing the structural relationship.

If no related cases exist at time of writing, state this explicitly and identify candidates for future Dispatch work.

### 2.14 Related Blog Primers

Links to 1–3 Blog articles that provide foundational context for readers less familiar with the framework or the market wave.

Format:
- [Blog Title](/blog/slug/): One sentence on what foundational concept the article covers.

### 2.15 PDF Dossier Availability

A section declaring the PDF status of this dossier.

When PDF is not yet available: state "PDF version pending. See Section 6 for PDF Dossier Standard."

When PDF is available: provide download link, version date, and a note that the PDF reflects the classification as of a specific date.

## 3. Source Governance

### Source Tiers

All factual claims in a Dispatch dossier must be assigned a source tier.

**Tier 1 — Primary / Authoritative**
- Company filings (annual reports, 10-K, 20-F, proxy statements)
- Official company pages (investor relations, product documentation)
- Official government and regulatory documents
- Court filings and legal records
- Standards body publications (RFC, ISO, IEEE, W3C)

Tier 1 sources are the highest-quality evidence available. Claims supported only by Tier 1 sources may be labeled **High Confidence**.

**Tier 2 — Recognized Institutional**
- Reputable industry bodies and trade associations
- Recognized academic and research institutions (peer-reviewed publications)
- Official regulatory body guidance and research
- Central bank and intergovernmental organization publications

Tier 2 sources support **High Confidence** or **Moderate Confidence** labeling depending on claim type.

**Tier 3 — Reputable Secondary**
- Established financial press (Financial Times, Wall Street Journal, Reuters, Bloomberg)
- Established technology press (Ars Technica, The Verge, Wired — for technical claims)
- Established industry trade media with editorial standards

Tier 3 sources support **Moderate Confidence** labeling. They may not be used as sole support for structural claims of significant magnitude.

**Tier 4 — Commentary and Secondary Interpretation**
- Analyst commentary, sell-side research, industry reports without primary backing
- Blog posts and opinion pieces, including from recognized commentators
- Aggregated or summary data where primary sources are not cited

Tier 4 sources must be labeled explicitly and used only when no higher-tier source is available. Claims resting only on Tier 4 sources should be labeled **Watch Signal** or **Unverified**.

### Source Rules

1. **No fragile current claims without source support.** Any claim about current market share, revenue, customer count, or operational scale must include a source and date.

2. **No future project claims without source backing.** Roadmap claims, announced products, strategic plans, and expansion projects must be source-attributed with a date-checked reference.

3. **No geopolitical claims without Tier 1 or Tier 2 backing.** Regulatory, trade, sanctions, or policy claims must come from official sources.

4. **Facts and interpretation must be separated.** Analyst interpretation must be clearly labeled as such. The distinction between "sourced fact" and "analyst inference" must be visible to the reader.

5. **Mark uncertainty explicitly.** Do not reduce or soften uncertainty for readability. A dossier with explicit uncertainty is more valuable than one with false confidence.

6. **Date-check all current claims.** Claims about the present state of an actor's position, strategy, or market must include the date they were verified. Do not carry forward unchecked claims through dossier revisions.

## 4. Confidence Labels

Every significant claim in a Dispatch dossier must be assigned a confidence label where the evidence basis is not self-evident.

### Label Definitions

**High Confidence**  
The claim is directly supported by Tier 1 or Tier 2 sources. The evidence is specific, verifiable, and not contested by other credible sources. The analyst has no material reason to doubt the claim.

*Usage:* Use for foundational historical facts, regulatory records, company-disclosed operational data, and documented technical specifications.

**Moderate Confidence**  
The claim is supported by Tier 2 or Tier 3 sources, or by multiple corroborating Tier 3 sources. The evidence is credible but may involve inference, aggregation, or secondary reporting that could introduce inaccuracy.

*Usage:* Use for market dynamics, competitive positioning assessments, and technology adoption claims.

**Watch Signal**  
The claim is plausible and directionally supported, but lacks sufficient source backing to assert with confidence. It represents a signal worth tracking but not a verified fact.

*Usage:* Use for forward signals, emerging structural dynamics, and claims that are directionally supported but not confirmed. Do not present Watch Signals as conclusions.

**Unverified — Do Not Publish**  
The claim cannot be adequately supported at time of writing. It must not appear in a published dossier. If it is critical to the analysis, its absence must be stated explicitly in the Limits section.

**Requires Source Before Publication**  
The claim is structurally important but currently unsupported. A specific source must be located and verified before this section can be published. This label is used in draft stages only.

## 5. Future Project Rules

Future projects, strategic initiatives, and forward-looking claims may be included in a Dispatch dossier only when all of the following conditions are met:

1. **Publicly announced.** The initiative has been formally announced by the actor itself, by a regulatory or government body, or by a standards organization — not reported as rumor, leak, or analyst projection.

2. **Source-backed.** A specific, citable source exists for the announcement. The source must be retrievable at the time of publication and included in the Source-Governed Fact Table.

3. **Date-checked.** The source must be checked for the date of the announcement and the expected timeline of the initiative. The dossier must note when the claim was verified.

4. **Clearly distinguished from speculation.** The text must make unambiguous that this is a disclosed initiative, not analyst inference about likely future behavior.

When forward signals exist that do not meet these four requirements, they may be mentioned in the Limits and Uncertainty section as items to watch, clearly labeled as **Watch Signal — unconfirmed**.

Do not invent forward momentum to make a dossier appear more complete or timely.

## 6. PDF Dossier Standard

Each mature Dispatch dossier should eventually offer a downloadable PDF version. The PDF is a governed artifact with the same source-governance requirements as the HTML dossier.

### PDF Required Elements

Every PDF dossier must include:

| Element | Requirement |
|---|---|
| Dispatch number | Displayed on cover and in header |
| Canonical URL | Printed on cover (https://shovelssale.com/dispatch/NNN.html) |
| Version / date | Dossier version and the date of last review |
| Source posture | A brief statement of the source governance standard applied |
| Not-investment-advice note | Full declaration (verbatim from Section 2.12) |
| Citation and source section | All Tier 1–4 sources cited in the dossier, formatted consistently |
| Print-safe layout | No JavaScript-dependent content; readable in black-and-white print |

### PDF Governance Rules

- A PDF dossier is a point-in-time snapshot. It must include the date it was generated.
- When the HTML version of the dossier is updated, the PDF must be regenerated or flagged as outdated.
- PDF files must not be committed to the repository without a corresponding HTML update.
- PDF availability is a maturity signal, not a launch requirement. A new Dispatch may be published as HTML-only and upgraded to include a PDF in a subsequent governance cycle.

### PDF Scope

PDFs are appropriate when:
- The dossier meets the full structure requirements in Section 2
- The source-governed fact table has a minimum of 10 entries
- The dossier has been reviewed for language and classification accuracy
- The PDF does not contain any Unverified or "Requires Source Before Publication" claims

## 7. SEO and Internal Linking

### Required Links — Every Dispatch

Every published Dispatch must include the following internal links:

| Destination | Purpose |
|---|---|
| `/scanner/` | Direct readers to classify related actors |
| `/framework/` | Provide foundational framework context |
| `/dispatch/` | Link back to the Dispatch index |
| Related prior Dispatch pages | Structural cross-reference (see Section 2.13) |
| Related Blog primers | Foundational reading for non-expert visitors (see Section 2.14) |

### Link Placement Rules

- Internal links must be contextual, not mechanical. A link to `/scanner/` belongs in a sentence about classification, not at the bottom of the page as a list of links.
- Related Dispatch links must include a one-sentence structural connection. Do not link without explaining why the connection is structurally relevant.
- Blog primer links should appear in a contextually appropriate section, not pasted as a footer appendage.
- Do not link-farm. Every link must serve the reader's understanding.

### Reverse Linking — Blog Articles

When a Blog article covers a market wave, company, or structural concept that is also the subject of a Dispatch dossier, the Blog article should link to that Dispatch — contextually, where the connection adds analytical depth. This is not a mechanical requirement: only link when it genuinely strengthens the reader's understanding.

### SEO Technical Requirements

Every Dispatch page must satisfy:
- Exactly one `<h1>` element containing the Dispatch title
- A unique `<meta name="description">` that accurately describes the dossier's subject and structural classification
- A unique, accurate `<title>` tag (under 70 characters when possible)
- A canonical `<link>` to the fully qualified HTTPS URL (`https://shovelssale.com/dispatch/NNN.html`)
- Inclusion in `sitemap.xml` with appropriate priority and last-modified date
- No broken internal links (verified by `validate_links.py`)

## 8. Prohibited Content

The following content types are prohibited in any Dispatch dossier published on ShovelsSale.com.

### Investment Language

- No investment recommendations of any kind
- No buy, sell, or hold language
- No price targets, valuation estimates, or return projections
- No language implying that a structural classification predicts financial performance

### Hype Framing

- No superlatives used as structural claims without evidence ("the most important," "the dominant force," "unstoppable")
- No breathless tone or excitement-driven framing
- No language designed to manufacture urgency rather than convey structural insight

### Unsourced Strategic Claims

- No market share claims without a specific source and date
- No revenue or customer count claims without a source
- No competitive positioning claims that are not either sourced or labeled as analyst interpretation
- No roadmap claims presented as fact without meeting the Future Project Rules in Section 5

### Thin Content

- No generic company descriptions that could be found on the company's own website and add no structural analysis
- No short summaries presented as dossiers
- No padding through repetition or restatement without analytical development

### Fake Certainty

- No false precision ("this company controls 73.4% of X" without a source with that specific figure)
- No collapsing of Watch Signals into conclusions
- No omission of known uncertainty to appear more authoritative

### AI-Generated Filler

- No templated language used to fill required sections without genuine analytical content
- No generic boilerplate that is structurally equivalent across multiple dossiers
- No language that sounds analytical but makes no falsifiable structural claim

### Link Farming

- No affiliate links
- No links included for SEO value without genuine contextual relevance
- No external links to commercial services without disclosure

## 9. Quality Gate Alignment

Every Dispatch dossier must pass the following before publication or commit.

### Technical Requirements

| Check | Tool | Standard |
|---|---|---|
| HTML structure | Manual review | Exactly one `<h1>` |
| Canonical URL | Manual review | Present and correct HTTPS URL |
| Meta title | Manual review | Present, unique, under 70 characters when possible |
| Meta description | Manual review | Present, unique, accurate |
| Internal link integrity | `validate_links.py` | No broken internal links |
| Asset integrity | `validate_assets.py` | No missing local assets |
| Sitemap coverage | `update_sitemap.py` + `validate_seo.py` | Page included in sitemap |
| Content integrity | `validate_content.py` | No placeholder content detected |
| Full quality gate | `quality_gate.py` | Exit code 0 |

### Content Requirements

Before publication, the author must verify:

- [ ] All sections from Section 2 are present (or explicitly note why a section cannot be completed)
- [ ] Source-Governed Fact Table contains at least 5 entries
- [ ] All confidence labels are assigned where required
- [ ] Future project claims meet all four Future Project Rules
- [ ] Prohibited content checklist passed
- [ ] Not-investment-advice declaration is present
- [ ] Related Dispatch and Blog links are contextual, not mechanical
- [ ] All internal links resolve correctly

### Scanner Alignment

- The Classification Summary must include scanner scores that can be reproduced using the Shovel Scanner at `/scanner/`
- If a preset exists for the actor, the preset values and classification must be documented
- If the scanner classification differs from the analyst's assessment, the divergence must be explained in Section 2.9 (Scanner Interpretation)
